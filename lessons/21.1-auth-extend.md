# Lesson 21.1: Extending Auth -- Forgot Password, Change Password, Email Verification (OTP)

## What You Will Learn

- Rate-limiting and log-hygiene basics so OTP endpoints don't become a spam vector or an information leak

## 21.1.1 The Big Picture

Lesson 20/21 gave us `POST /auth/register`, `POST /auth/login`, and `GET /auth/me`. That is enough to prove identity, but not to **recover** an account when a user forgets their password, or to **verify** that they actually own the email address they signed up with.

The industry-standard answer for both is a **one-time password** (OTP): a short numeric code, sent out-of-band (via email/SMS/authenticator app), that expires quickly and can only be used once. We are using email here because we already have an email column and Mailtrap gives us a free sandbox inbox.

```
Forgot password (public)
  1) POST /auth/forgot-password   { email }
       -> server generates a 6-digit OTP, hashes it, saves it against the user,
          emails the plaintext code to the user's inbox
  2) POST /auth/reset-password    { email, otp, newPassword }
       -> server verifies the hashed OTP, checks it hasn't expired,
          marks it consumed, hashes and stores the new password

Change password (authenticated)
  POST /auth/change-password       { currentPassword, newPassword }
    -> requires a valid JWT; verifies currentPassword before overwriting

Verify email (authenticated)
  1) POST /auth/send-email-verify-otp     (no body needed -- we use the JWT's user id)
       -> generate a 6-digit OTP, hash + save, email it
  2) POST /auth/verify-email              { otp }
       -> verify the OTP; if good, set user.emailVerified = true
```

Every flow above uses the **same underlying `OtpToken` model + `otpService`** -- so once you understand the first one, you understand all three.

**Why not send the plaintext code back in the API response?** Because that would defeat the point: the code has to arrive through a *second* channel (the inbox) so an attacker with only the frontend can't complete the flow. The API just says "sent". If the email address is unknown, it still says "sent" -- so we never leak which emails have accounts.

## 21.1.4 Nodemailer Transporter

We create the transporter once at boot and share it across the app.

```ts
// booking-backend/src/config/mail.ts
import nodemailer, { Transporter } from "nodemailer";

interface MailConfig {
  host: string;
  port: number;
  secure: boolean;
  auth: { user: string; pass: string };
  from: string;
}

function readMailConfig(): MailConfig {
  const host = process.env.SMTP_HOST;
  const user = process.env.SMTP_USERNAME;
  const pass = process.env.SMTP_PASSWORD;
  const from = process.env.SMTP_DEFAULT_FROM;

  if (!host || !user || !pass || !from) {
    throw new Error(
      "Mail config missing. Set SMTP_HOST, SMTP_USERNAME, SMTP_PASSWORD and SMTP_DEFAULT_FROM in .env"
    );
  }

  return {
    host,
    port: Number(process.env.SMTP_PORT) || 587,
    secure: (process.env.MAIL_SECURE ?? "false").toLowerCase() === "true",
    auth: { user, pass },
    from,
  };
}

// Lazy-init pattern: only construct the transporter the first time we need it.
// This means routes that don't send email don't force a bad SMTP config to fail
// at boot -- important during local development.
let cached: { transporter: Transporter; from: string } | null = null;

export function getMailer(): { transporter: Transporter; from: string } {
  if (cached) return cached;
  const cfg = readMailConfig();
  const transporter = nodemailer.createTransport({
    host: cfg.host,
    port: cfg.port,
    secure: cfg.secure,
    auth: cfg.auth,
  });
  cached = { transporter, from: cfg.from };
  return cached;
}
```

**Why `secure: false` on port 587?** Port 587 uses STARTTLS: the connection begins in plaintext and is upgraded to TLS mid-handshake. Nodemailer's `secure: true` means "start TLS immediately" and is for port 465. Getting these mixed up is the #1 reason SMTP hangs.

## 21.1.5 Sending Email

Wrap the transporter in a small helper so the rest of the app never has to know about MIME parts or Nodemailer's API.

```ts
// booking-backend/src/services/mailService.ts
import { getMailer } from "../config/mail";

interface SendMailOptions {
  to: string;
  subject: string;
  html: string;
  text?: string;
}

export async function sendMail(opts: SendMailOptions): Promise<void> {
  const { transporter, from } = getMailer();
  await transporter.sendMail({
    from,
    to: opts.to,
    subject: opts.subject,
    html: opts.html,
    // Nodemailer generates a text fallback from the HTML if we don't
    // provide one, but supplying our own lands better with strict clients.
    text: opts.text ?? stripHtml(opts.html),
  });
}

function stripHtml(html: string): string {
  return html
    .replace(/<br\s*\/?>/gi, "\n")
    .replace(/<\/(p|div|h[1-6])>/gi, "\n")
    .replace(/<[^>]+>/g, "")
    .replace(/\s+\n/g, "\n")
    .trim();
}

export function otpEmail(params: {
  name: string;
  code: string;
  purpose: "password_reset" | "email_verify";
  expiresInMinutes: number;
}): { subject: string; html: string } {
  const heading =
    params.purpose === "password_reset"
      ? "Reset your BookMyRoom password"
      : "Verify your BookMyRoom email";
  const subject =
    params.purpose === "password_reset"
      ? "Your password reset code"
      : "Your email verification code";
  const action =
    params.purpose === "password_reset"
      ? "reset your password"
      : "verify your email address";

  const html = `
    <div style="font-family: system-ui, sans-serif; max-width: 480px; margin: 0 auto; padding: 24px; color: #111;">
      <h2 style="margin-top:0">${heading}</h2>
      <p>Hi ${escape(params.name)},</p>
      <p>Use the code below to ${action}. It expires in <strong>${params.expiresInMinutes} minutes</strong> and can only be used once.</p>
      <div style="font-size: 32px; letter-spacing: 8px; font-weight: bold; text-align: center; padding: 24px; background: #f6f6f6; border-radius: 8px; margin: 24px 0;">
        ${params.code}
      </div>
      <p style="color:#666; font-size: 13px;">If you didn't request this, you can safely ignore this email.</p>
    </div>
  `;
  return { subject, html };
}

// Minimal HTML escape -- never interpolate raw user input into an email body.
function escape(s: string): string {
  return s
    .replaceAll("&", "&amp;")
    .replaceAll("<", "&lt;")
    .replaceAll(">", "&gt;")
    .replaceAll('"', "&quot;")
    .replaceAll("'", "&#39;");
}
```

**Why HTML-escape the name?** Because `user.name` came from a form. If a malicious user set their name to `<script>...</script>`, the email client might execute it. Escaping is a one-line safety net.

## 21.1.6 The `OtpToken` Model

We store OTPs in a **separate collection** rather than fields on `User`. Three reasons:

1. **Purpose isolation.** A password-reset code should never be usable to verify an email, and vice versa. A `purpose` field makes that explicit and enforceable.
2. **Multiple pending codes are natural.** A user might request an OTP, mistype it, and request another. The old one should be expired without touching the user document.
3. **Housekeeping is one query.** `db.otpTokens.deleteMany({ expiresAt: { $lt: now } })` -- easy to schedule if you ever add a cron.

```ts
// booking-backend/src/models/OtpToken.ts
import mongoose, { Schema, Document, Types } from "mongoose";

export type OtpPurpose = "password_reset" | "email_verify";

export interface IOtpToken extends Document {
  user: Types.ObjectId;
  purpose: OtpPurpose;
  tokenHash: string;      // SHA-256 hash of the OTP -- we never store plaintext
  expiresAt: Date;
  consumedAt?: Date;      // set when the OTP has been successfully used
  attempts: number;       // failed-verification counter to defuse brute force
  createdAt: Date;
  updatedAt: Date;
}

const otpTokenSchema = new Schema<IOtpToken>(
  {
    user: { type: Schema.Types.ObjectId, ref: "User", required: true, index: true },
    purpose: {
      type: String,
      enum: ["password_reset", "email_verify"],
      required: true,
    },
    tokenHash: { type: String, required: true },
    expiresAt: { type: Date, required: true, index: true },
    consumedAt: { type: Date, default: undefined },
    attempts: { type: Number, default: 0 },
  },
  { timestamps: true }
);

// TTL index -- MongoDB deletes documents automatically after expiresAt.
// The `expireAfterSeconds: 0` means "as soon as the document's own
// expiresAt is in the past", which is what we want.
otpTokenSchema.index({ expiresAt: 1 }, { expireAfterSeconds: 0 });

// Fast lookup for "does this user have a live OTP for this purpose?"
otpTokenSchema.index({ user: 1, purpose: 1, consumedAt: 1 });

const OtpToken = mongoose.model<IOtpToken>("OtpToken", otpTokenSchema);
export default OtpToken;
```

> **Never store the plaintext OTP.** If your database is ever leaked, an attacker who reads plaintext OTPs can hijack any pending reset. Hashing means the leak is worthless: a 6-digit code has only 1,000,000 possibilities but the attacker cannot use them without also having your `SHA-256` inputs *and* being within the 10-minute window *and* knowing the target user *and* not tripping the attempt counter. Cost per attempt matters.

## 21.1.7 The `otpService`

One place that knows how to *issue*, *hash*, and *verify* an OTP. Every controller calls into this -- no controller writes to `OtpToken` directly.

```ts
// booking-backend/src/services/otpService.ts
import crypto from "crypto";
import { Types } from "mongoose";
import OtpToken, { IOtpToken, OtpPurpose } from "../models/OtpToken";

const OTP_TTL_MINUTES = 10;
const MAX_ATTEMPTS = 5;
const OTP_LENGTH = 6;

export function otpTtlMinutes(): number {
  return OTP_TTL_MINUTES;
}

function generateNumericCode(length: number): string {
  // crypto.randomInt is uniform (unlike Math.random) so all codes are equally likely.
  let code = "";
  for (let i = 0; i < length; i++) code += crypto.randomInt(0, 10).toString();
  return code;
}

function hashCode(code: string): string {
  // SHA-256 is fine here: OTPs are high-entropy short-lived secrets, and we
  // want verify() to run in a couple of microseconds, not the tens of
  // milliseconds bcrypt would cost per attempt.
  return crypto.createHash("sha256").update(code).digest("hex");
}

// Issue a new OTP: invalidates any prior OTP for the same (user, purpose)
// pair, then creates + returns the fresh plaintext code so the caller can
// send it by email. The plaintext never leaves the service after this.
export async function issueOtp(
  userId: string | Types.ObjectId,
  purpose: OtpPurpose
): Promise<string> {
  const now = new Date();

  await OtpToken.updateMany(
    { user: userId, purpose, consumedAt: { $exists: false } },
    { $set: { consumedAt: now } }
  );

  const code = generateNumericCode(OTP_LENGTH);
  await OtpToken.create({
    user: userId,
    purpose,
    tokenHash: hashCode(code),
    expiresAt: new Date(now.getTime() + OTP_TTL_MINUTES * 60 * 1000),
    attempts: 0,
  });

  return code;
}

interface VerifyResult {
  ok: boolean;
  reason?: "not_found" | "expired" | "too_many_attempts" | "mismatch";
}

// Verify + consume: if the code is right, the token is marked consumed
// atomically. If it's wrong, the attempt counter goes up. On MAX_ATTEMPTS,
// the token is invalidated even if the caller keeps guessing.
export async function verifyOtp(
  userId: string | Types.ObjectId,
  purpose: OtpPurpose,
  code: string
): Promise<VerifyResult> {
  const now = new Date();
  const token: IOtpToken | null = await OtpToken.findOne({
    user: userId,
    purpose,
    consumedAt: { $exists: false },
  }).sort({ createdAt: -1 });

  if (!token) return { ok: false, reason: "not_found" };
  if (token.expiresAt < now) return { ok: false, reason: "expired" };
  if (token.attempts >= MAX_ATTEMPTS) {
    return { ok: false, reason: "too_many_attempts" };
  }

  if (token.tokenHash !== hashCode(code)) {
    token.attempts += 1;
    if (token.attempts >= MAX_ATTEMPTS) token.consumedAt = now;
    await token.save();
    return { ok: false, reason: "mismatch" };
  }

  token.consumedAt = now;
  await token.save();
  return { ok: true };
}
```

**Design notes:**

- Every new OTP consumes the previous one. Users often click "Resend" -- we do not want two live codes lying around.
- `verifyOtp` reads the most recent non-consumed token, not "the one matching this hash". That means a wrong guess still increments the counter for the *correct* pending code, which is what we want.
- Consuming on `MAX_ATTEMPTS` means an attacker has exactly 5 guesses per issued code. Combined with 6 digits, that's 5-in-a-million per code, per user.

## 21.1.9 Forgot Password: Request OTP

```ts
// booking-backend/src/controllers/authController.ts  (append)
import { issueOtp, verifyOtp, otpTtlMinutes } from "../services/otpService";
import { sendMail, otpEmail } from "../services/mailService";

// POST /api/auth/forgot-password
export const forgotPassword = async (req: Request, res: Response): Promise<void> => {
  try {
    const { email } = req.body as { email: string };

    // Same response whether or not the email exists -- we never leak
    // which addresses have accounts. Attackers use these endpoints to
    // enumerate users; symmetric responses stop that.
    const user = await User.findOne({ email });
    if (user) {
      const code = await issueOtp(user.id, "password_reset");
      const { subject, html } = otpEmail({
        name: user.name,
        code,
        purpose: "password_reset",
        expiresInMinutes: otpTtlMinutes(),
      });
      // Await the send so failures surface to the client. In production
      // you might queue this via a background worker instead.
      await sendMail({ to: user.email, subject, html });
    }

    res.status(200).json({
      message: "If an account exists for that email, a code has been sent.",
    });
  } catch (error: unknown) {
    console.error("forgotPassword error:", error);
    res.status(500).json({ message: "Failed to send password reset code" });
  }
};
```

**Do not `console.log(code)`.** During development you can peek at the Mailtrap inbox, or add a temporary `console.log` behind a `NODE_ENV === "development"` check. Never log OTPs on the same server as your production database.

## 21.1.10 Reset Password: Verify OTP + Set New Password

We do this in a **single endpoint** for teaching simplicity. The wizard on the frontend hides the two steps behind one form submission.

```ts
// POST /api/auth/reset-password
export const resetPassword = async (req: Request, res: Response): Promise<void> => {
  try {
    const { email, otp, newPassword } = req.body as {
      email: string;
      otp: string;
      newPassword: string;
    };

    const user = await User.findOne({ email });
    if (!user) {
      // Still symmetric -- do not confirm existence to a client that
      // reached this step without a valid code.
      res.status(400).json({ message: "Invalid or expired code" });
      return;
    }

    const result = await verifyOtp(user.id, "password_reset", otp);
    if (!result.ok) {
      const map: Record<string, string> = {
        not_found: "Invalid or expired code",
        expired: "Invalid or expired code",
        too_many_attempts: "Too many attempts. Request a new code and try again.",
        mismatch: "Invalid or expired code",
      };
      res.status(400).json({ message: map[result.reason!] });
      return;
    }

    user.password = await bcrypt.hash(newPassword, 10);
    await user.save();

    res.status(200).json({
      message: "Password reset successful. You can now sign in with your new password.",
    });
  } catch (error: unknown) {
    console.error("resetPassword error:", error);
    res.status(500).json({ message: "Failed to reset password" });
  }
};
```

**Same generic message** for `not_found`, `expired`, and `mismatch`. From the attacker's point of view "wrong code" and "no such user" have to be indistinguishable.

## 21.1.11 Change Password (Authenticated)

```ts
// POST /api/auth/change-password
export const changePassword = async (req: Request, res: Response): Promise<void> => {
  try {
    const { currentPassword, newPassword } = req.body as {
      currentPassword: string;
      newPassword: string;
    };
    const userId = req.user!.userId;

    // We deliberately re-select the password since the schema hides it by default.
    const user = await User.findById(userId).select("+password");
    if (!user) {
      res.status(404).json({ message: "User not found" });
      return;
    }

    const ok = await bcrypt.compare(currentPassword, user.password);
    if (!ok) {
      res.status(400).json({ message: "Current password is incorrect" });
      return;
    }

    if (currentPassword === newPassword) {
      res
        .status(400)
        .json({ message: "New password must be different from current password" });
      return;
    }

    user.password = await bcrypt.hash(newPassword, 10);
    await user.save();

    res.status(200).json({ message: "Password changed successfully" });
  } catch (error: unknown) {
    console.error("changePassword error:", error);
    res.status(500).json({ message: "Failed to change password" });
  }
};
```

Note that we do **not** rotate the JWT after a password change. That's a design choice we can make later (blacklist old tokens, or embed a `passwordChangedAt` claim). For a teaching lesson, the current session stays valid.

## 21.1.17 Frontend: React Query Mutation Hooks

```ts
// booking-frontend/src/hooks/useAuth.ts   (append -- alongside existing useLogin etc.)
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { toast } from "sonner";
import { authApi } from "@/services/authApi";
import { authKeys } from "@/hooks/useAuth";
import type {
  ForgotPasswordData,
  ResetPasswordData,
  ChangePasswordData,
  VerifyEmailData,
} from "@/types/user";

export function useForgotPassword() {
  return useMutation({
    mutationFn: (payload: ForgotPasswordData) => authApi.forgotPassword(payload),
    onSuccess: (data) => toast.success(data.message),
    onError: (error: Error) =>
      toast.error(error.message || "Failed to send reset code"),
  });
}

export function useResetPassword() {
  return useMutation({
    mutationFn: (payload: ResetPasswordData) => authApi.resetPassword(payload),
    onSuccess: (data) => toast.success(data.message),
    onError: (error: Error) =>
      toast.error(error.message || "Failed to reset password"),
  });
}

export function useChangePassword() {
  return useMutation({
    mutationFn: (payload: ChangePasswordData) => authApi.changePassword(payload),
    onSuccess: (data) => toast.success(data.message),
    onError: (error: Error) =>
      toast.error(error.message || "Failed to change password"),
  });
}

export function useSendEmailVerifyOtp() {
  return useMutation({
    mutationFn: () => authApi.sendEmailVerifyOtp(),
    onSuccess: (data) => toast.success(data.message),
    onError: (error: Error) =>
      toast.error(error.message || "Failed to send verification code"),
  });
}

export function useVerifyEmail() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (payload: VerifyEmailData) => authApi.verifyEmail(payload),
    onSuccess: (data) => {
      toast.success(data.message);
      // Cheapest way to refresh emailVerified in the app: invalidate the
      // current-user query so the header, profile page, etc. re-fetch.
      qc.invalidateQueries({ queryKey: authKeys.user() });
    },
    onError: (error: Error) =>
      toast.error(error.message || "Failed to verify email"),
  });
}
```

> **Two hooks in the same file -- OK to `import ... from "@/hooks/useAuth"` recursively?** Yes: the import is a *type-and-symbol* reference that resolves after the file is compiled. Some teams still prefer to move `authKeys` into `hooks/authKeys.ts` for clarity. Either works.

## 21.1.18 Frontend: The Forgot Password Wizard

Two-step flow on a single page. Step 1 collects the email and requests the code. Step 2 collects OTP + new password and submits the reset.

```tsx
// booking-frontend/src/pages/ForgotPasswordPage.tsx
import { useState } from "react";
import { Controller, useForm } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";
import { z } from "zod";
import { Link, useNavigate } from "react-router-dom";
import {
  Field,
  FieldError,
  FieldGroup,
  FieldLabel,
} from "@/components/ui/field";
import { Input } from "@/components/ui/input";
import { Button } from "@/components/ui/button";
import { useForgotPassword, useResetPassword } from "@/hooks/useAuth";

const emailSchema = z.object({
  email: z.string().email("Please enter a valid email"),
});
type EmailForm = z.infer<typeof emailSchema>;

const resetSchema = z.object({
  otp: z.string().regex(/^\d{6}$/, "Enter the 6-digit code"),
  newPassword: z.string().min(6, "Password must be at least 6 characters"),
});
type ResetForm = z.infer<typeof resetSchema>;

export function ForgotPasswordPage() {
  const navigate = useNavigate();
  const [step, setStep] = useState<"email" | "reset">("email");
  const [email, setEmail] = useState<string>("");

  const forgot = useForgotPassword();
  const reset = useResetPassword();

  const emailForm = useForm<EmailForm>({
    resolver: zodResolver(emailSchema),
    defaultValues: { email: "" },
  });

  const resetForm = useForm<ResetForm>({
    resolver: zodResolver(resetSchema),
    defaultValues: { otp: "", newPassword: "" },
  });

  const submitEmail = (values: EmailForm) => {
    forgot.mutate(values, {
      onSuccess: () => {
        setEmail(values.email);
        setStep("reset");
      },
    });
  };

  const submitReset = (values: ResetForm) => {
    reset.mutate(
      { email, otp: values.otp, newPassword: values.newPassword },
      { onSuccess: () => navigate("/login") }
    );
  };

  return (
    <div className="mx-auto w-full max-w-sm space-y-6">
      <div>
        <h1 className="text-2xl font-bold">Forgot your password?</h1>
        <p className="text-muted-foreground text-sm">
          {step === "email"
            ? "Enter your account email and we'll send you a 6-digit code."
            : `We've sent a code to ${email}. Enter it below to choose a new password.`}
        </p>
      </div>

      {step === "email" ? (
        <form onSubmit={emailForm.handleSubmit(submitEmail)} className="space-y-6">
          <FieldGroup>
            <Controller
              name="email"
              control={emailForm.control}
              render={({ field, fieldState }) => (
                <Field data-invalid={fieldState.invalid}>
                  <FieldLabel htmlFor={field.name}>Email</FieldLabel>
                  <Input
                    {...field}
                    id={field.name}
                    type="email"
                    placeholder="you@example.com"
                    aria-invalid={fieldState.invalid}
                  />
                  {fieldState.invalid && (
                    <FieldError errors={[fieldState.error]} />
                  )}
                </Field>
              )}
            />
          </FieldGroup>
          <Button type="submit" className="w-full" disabled={forgot.isPending}>
            {forgot.isPending ? "Sending..." : "Send reset code"}
          </Button>
        </form>
      ) : (
        <form onSubmit={resetForm.handleSubmit(submitReset)} className="space-y-6">
          <FieldGroup>
            <Controller
              name="otp"
              control={resetForm.control}
              render={({ field, fieldState }) => (
                <Field data-invalid={fieldState.invalid}>
                  <FieldLabel htmlFor={field.name}>6-digit code</FieldLabel>
                  <Input
                    {...field}
                    id={field.name}
                    inputMode="numeric"
                    maxLength={6}
                    placeholder="123456"
                    aria-invalid={fieldState.invalid}
                  />
                  {fieldState.invalid && (
                    <FieldError errors={[fieldState.error]} />
                  )}
                </Field>
              )}
            />
            <Controller
              name="newPassword"
              control={resetForm.control}
              render={({ field, fieldState }) => (
                <Field data-invalid={fieldState.invalid}>
                  <FieldLabel htmlFor={field.name}>New password</FieldLabel>
                  <Input
                    {...field}
                    id={field.name}
                    type="password"
                    placeholder="At least 6 characters"
                    aria-invalid={fieldState.invalid}
                  />
                  {fieldState.invalid && (
                    <FieldError errors={[fieldState.error]} />
                  )}
                </Field>
              )}
            />
          </FieldGroup>
          <Button type="submit" className="w-full" disabled={reset.isPending}>
            {reset.isPending ? "Resetting..." : "Reset password"}
          </Button>
          <Button
            type="button"
            variant="ghost"
            className="w-full"
            onClick={() => setStep("email")}
          >
            Use a different email
          </Button>
        </form>
      )}

      <p className="text-muted-foreground text-center text-sm">
        Remembered it? <Link to="/login" className="underline">Sign in</Link>
      </p>
    </div>
  );
}
```

## 21.1.21 Security Considerations Worth Teaching

The endpoints above are safe *on this teaching stack*. On the way to production you would add each of the following. It is worth showing students where each hook goes even if you don't implement them today.

| Concern | Fix |
|--------|------|
| **Email enumeration** | Already handled: `/forgot-password` and `/reset-password` return the same messages regardless of whether the email exists. |
| **OTP spam** | Add `express-rate-limit` to `/forgot-password`, `/send-email-verify-otp` and the verify endpoints. Something like `windowMs: 15 * 60_000, max: 5` per IP. |
| **Brute-force guessing** | `otpService.MAX_ATTEMPTS = 5` handles per-token. Add an IP + email lockout after N failed verify calls if you want an extra belt. |
| **OTP replay** | `consumedAt` on `OtpToken` + the "invalidate previous on issue" behaviour in `issueOtp` blocks replay. |
| **Timing attacks on OTP compare** | For a hashed 6-digit code, a hex-string compare is not the classic timing hazard, but if you ever move to longer secrets prefer `crypto.timingSafeEqual`. |
| **Log hygiene** | Never log the plaintext OTP, the JWT, or the user's password. `console.error(err)` on catch is fine because the error object does not contain those in this codebase. |
| **JWT rotation on password change** | Optional but common: embed `passwordChangedAt` in the User and reject any JWT issued before that time in `requireAuth`. Force re-login after a successful `changePassword` / `resetPassword`. |
| **Email delivery outage** | Await `sendMail` and let the client know if it failed. In production, queue via a background worker (BullMQ, SQS) so a slow SMTP does not block the API. |

## Key Takeaways
1. **OTPs are hashed and single-use.** Never store the plaintext, never re-use the same code across purposes, never issue a second live code without consuming the first.
2. **Symmetric responses.** Public flows must return the same message whether or not the email exists. Attackers use these endpoints to enumerate users.
3. **One service, many controllers.** `issueOtp` / `verifyOtp` live in `services/otpService.ts` and every controller calls into them. No controller writes to `OtpToken` directly.
4. **Separate collection, not extra User fields.** `OtpToken` isolates lifecycle, gets a TTL index for free, and keeps `User` focused on identity.
5. **Frontend hooks are terse.** Each mutation is a 6-line React Query hook that surfaces the server's message via Sonner. No component ever composes toast strings.
6. **Wizards, not multi-endpoint dances.** `ForgotPasswordPage` is a single URL with a `step` state. Users perceive one flow; the server sees two clean POSTs.
7. **Every OTP endpoint is a rate-limit target.** Even if you don't add the middleware today, know where it goes and why.
